AC23.02 — Développer une application à partir d’un cahier des charges

Développement d’une application web à partir d’un cahier des charges, avec maquettes, API et tests.

Objectif

Développer une application complète (SAE 3.02 “Search Server”) en respectant un cahier des charges : communication client/serveur, interface web et résultats fiables.

Compétences visées

  • Traduire un besoin en fonctionnalités concrètes.
  • Organiser le développement (maquettes, tâches, tests).
  • Mettre en place un front + un back simples mais propres.

Contexte du projet (SAE 3.02)

Projet “Search Server” : un service local de recherche de documents avec :

  • un serveur TCP multi‑thread,
  • un client TCP interactif,
  • une interface web Flask.

Le cahier des charges demandait une recherche propre et des résultats lisibles.

Fonctions clés :

  • mode phrase et mode termes (AND/OR),
  • recherche insensible à la casse,
  • recherche sur mot entier,
  • support regex.

Outils utilisés

  • Python 3 (sockets TCP)
  • Flask (interface web)
  • HTML/CSS (UI simple)
  • Git (versioning)

Méthodologie

  1. Découpage en trois briques : serveur TCP, client TCP, interface web.
  2. Tests unitaires simples avec des fichiers dans data/.
  3. Itérations UI pour rendre l’affichage clair.

    data_abs = os.path.abspath(DATA_PATH)
    if not candidate.startswith(data_abs + os.sep):
        abort(403)
    if not os.path.isfile(candidate):

    return candidate

Résultat

L’application répond aux exigences : recherche fonctionnelle en local, interface claire, et communication serveur fiable.

Ce que j’ai appris

  • La communication client/serveur doit être testée tôt pour éviter les blocages.
  • Une interface simple suffit si les résultats sont clairs.

Ce que j’ai fait

J’ai développé une application de recherche de documents à partir d’un cahier des charges, avec serveur TCP, client et interface web Flask.

Pourquoi je l’ai fait

Je l’ai fait pour passer d’un besoin exprimé en fonctionnalités à une solution cohérente, structurée et testable de bout en bout.

Comment je l’ai fait

  • Découpage du projet en trois briques : serveur, client et interface web.
  • Développement itératif avec tests de recherche et d’accès aux fichiers.
  • Ajustements de l’interface pour rendre les résultats compréhensibles.

Mes difficultés

La coordination entre les trois parties du projet a été le principal point de vigilance. Une interface propre ne suffit pas si le contrat entre les modules n’est pas stabilisé.

Ce que j’en ai appris

J’ai appris qu’un projet guidé par cahier des charges demande autant d’organisation que de code. Le plus important est d’assurer la cohérence des échanges entre composants.

Ce que je ferais autrement

Je définirais plus tôt un plan de tests fonctionnels et une mini spécification d’interface entre modules afin de limiter les ajustements tardifs.
